In the medical device world, clinicians suffer from "data fatigue"—they don't want to dig through 15 pages of normal pacing parameters just to find out a patient had a burst of VT or their lead impedance spiked. An "Exception-Only Top Sheet" acts as a clinical executive summary.

Here is a comprehensive plan for how we could structure and implement this feature:

### 1. Defining "Exceptions" (The Rules Engine)
We need to define what constitutes an exception. There are two primary types of exceptions we need to track:

*   **Absolute Thresholds:** The value is objectively abnormal, regardless of history.
    *   *Example:* Battery Status is "ERI" (Elective Replacement Indicator) or "EOL".
    *   *Example:* AT/AF Burden > 5%.
    *   *Example:* Any treated VT/VF episode count > 0.
*   **Relative/Delta Thresholds:** The value might be in a normal range, but the *change* since the last report is concerning.
    *   *Example:* RV Lead Impedance increased or decreased by more than 30% from the previous report.
    *   *Example:* RV Pacing Threshold increased by > 1.0V.
    *   *Example:* Biventricular (BiV) pacing dropped from 98% to 85% (indicating loss of CRT therapy).

### 2. Implementation Architecture

#### A. The Comparison Logic (Where does it live?)
We have two choices: Backend (Go) or Frontend (React). 
*   **Recommendation:** Build a lightweight utility on the **Frontend** first (e.g., `src/utils/clinicalRulesEngine.ts`). Since the frontend already fetches the `currentReport` and the `previousReport` (for pre-population), it has all the data it needs to run a fast comparison and generate an array of "Alert Objects" before rendering the UI or the PDF.

#### B. The Data Structure
The rules engine would output an array of standard objects, making it easy to render anywhere:
```typescript
interface ClinicalException {
  category: 'Battery' | 'Lead' | 'Arrhythmia' | 'Therapy';
  severity: 'Critical' | 'Warning' | 'Info';
  metric: string; // e.g., "RV Impedance"
  currentValue: string; // e.g., "1500 ohms"
  previousValue?: string; // e.g., "600 ohms"
  message: string; // e.g., "RV Impedance increased by 150% since last check."
}
```

#### C. User Interface (UI) Updates
1.  **Report Detail View:** At the very top of the `ReportDetail` page (before any of the standard data tables), we render an "Exception Summary" Card.
    *   If no exceptions are found, it shows a green banner: *"All parameters within normal limits or unchanged."*
    *   If exceptions are found, it lists them with color-coded badges (Red for Critical, Yellow for Warning).
2.  **During Data Entry (Form):** While the tech is typing in the `ReportForm`, if they enter an RV Impedance of 2000, a small inline warning could pop up: *"Warning: This is a 150% increase from the last report."* This catches typos instantly.

#### D. PDF Generation (The "Top Sheet")
When the user clicks "Download PDF" or generates the final signed report:
1.  Our `usePdfFormFiller` hook calls the rules engine.
2.  If there are exceptions, we use `pdf-lib` to dynamically generate a clean, single "Top Sheet" page.
3.  We insert this generated page as **Page 1** of the final merged PDF, pushing the raw manufacturer upload or standard form data to Page 2.

### 3. Phased Rollout Plan

If we decide to build this, here is how we'd break it down to keep it manageable:

*   **Phase 1: The Engine & Arrhythmias:** Build the basic rule engine focusing *only* on the most critical items: Battery ERI/EOL status and Arrhythmia counts (AF burden, shocks delivered). Display this at the top of the web UI.
*   **Phase 2: Deltas & Leads:** Add the mathematical logic to calculate percentage changes for Lead Impedances and Pacing Thresholds compared to the `previousReport`.
*   **Phase 3: The PDF Top Sheet:** Modify the PDF generator to prepend the physical Top Sheet page to the final document.
*   **Phase 4 (Future Enterprise Feature):** Allow the Clinic Admin to customize the threshold values (e.g., Dr. Smith considers AF Burden > 2% an exception, while Dr. Jones uses 5%).
